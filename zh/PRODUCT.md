# SteadySpec 产品目的与连续性合同

合同版本：2。描述符 schema 版本：1。状态：SteadySpec 0.x 系列的规范性
产品目的、当前参考架构与演进边界。

本合同取代保存在 [`docs/product-contract-history/v1/`](../docs/product-contract-history/v1/)
中的预发布 v1 合同。v1 正确阻止了五动词生命周期被静默降格，但把当前架构
放得过于接近产品的最终目的。

## 运行前提：责任来自外部关系

SteadySpec 用在重要工作被委托给 AI Agent，且 Agent 的产出吞吐可能已经超过
责任主体逐项重做或检查能力的场景。人类责任主体或组织仍然依据周围的社会、
组织或法律安排，拥有进入现实的权力并承担后果。

这种责任是外部运行前提，不是 SteadySpec 产生的结果。方法不创造、转移、
认证、履行或解除责任、法律义务、充分知情的批准或实际理解；它也不假定人
在技术上一定强于 Agent。

## 产品目的：目的保真下的能力兑现

SteadySpec 治理委托，以帮助 Agent：

- 忠实于已经授权的目的和约束；
- 在不静默修改目的的前提下，质疑可疑假设、问题框架和建议手段；
- 减少由不完整提问、过时经验、过早收敛到低上限方案、执行漂移和误导性
  收口造成的可避免能力损失；
- 让完成声明不超过已观察证据，并保留 unknown、fallback、debt 和残余风险；
- 以可扩展的人类注意力成本，把价值选择、风险接受、不可逆权力和最终验收
  交还给承担责任的人。

这是设计目标和未经验证的产品假设，不是对无偏、最优、正确、安全或高质量
结果的保证。

## 稳定核心原则

### PC-1：经授权目的的保真

Agent 必须区分已经授权的结果及硬约束，与 prompt 中的事实假设、问题框架和
建议实现手段。它可以质疑任何一层并建议重新打开；但未经明确批准或预先记录
的委托范围，不得采用对人类所有的目的、价值、风险或不可逆约束的修改。

当 prompt 不完整或内部冲突时，字面服从不等于目的保真；当目的仍不明确时，
推断不等于授权。

### PC-2：质疑而不越权

当更强替代方案、当前技术选项、反例或缺失专长可能显著改善结果时，Agent
应在承诺前把它们暴露出来。顺从不能静默压低答案上限。

质疑不授权单方面覆盖。与已确认目的或约束冲突时，必须先暴露、记录，并由
责任所有者解决，或确认处在既有委托边界内，才能继续执行。

### PC-3：避免过早收敛的能力兑现

SteadySpec 应在已声明的时间、成本、工具、证据、风险和授权约束内，减少
Agent 可用能力的可避免损失。治理得很干净但本可避免地弱，并不是期望结果。

没有流程能证明全局最优，也不能制造人与 Agent 都观察不到的现实。更多
debate、artifact 或算力并不自动更好。

### PC-4：证据边界内的声明完整性

测试、review、debate、cross-agent 共识、conformance 和人的接受都是有边界
的证据输入，不是语义真理。完成声明必须绑定精确候选和已观察覆盖；unknown、
fallback、debt 与残余风险不得被重写成 proof。

### PC-5：人的权力不是语义真理

在周围的权力关系下，承担责任的人拥有委托范围、价值排序、风险接受、不可逆
行动、最终验收、合并、部署和发布决定。Agent 可以承担大部分分析与执行，
也可以拥有有界、可逆的决策。

人的决定会在这段权责关系内授权行动，但不证明技术正确、理解完整或法律充分。

### PC-6：注意力路由是分诊，不是卸责

SteadySpec 不应要求人重做每一个机械步骤。它应把 must-read 决策、proof
边界、未满足意图、风险分类依据、unknown、debt、fallback 和 override path
交给承担责任的人，同时保留可审计底稿。

如果责任主体无法理解重要决策所需的基础，正确的下一步是更深入的 review、
寻找合适专家、缩小委托或停止，而不是生成更漂亮的摘要后做仪式性批准。

## 当前参考架构

上面的稳定核心是产品目的和评价边界。下面是当前规范手段：它们是受保护的
公共架构，不是哲学上永恒不变的目的。

### 可移植方法参考

[METHOD.md](METHOD.md) 中的八项机制及 rails-and-wings 责任/能力框架，是
当前领域中性的参考方法。如果所覆盖的失败模式、转换和证据边界仍然显式，
它们可以演进。

### 软件生命周期参考

当前规范且受兼容性保护的软件生命周期是：

```text
explore -> propose -> apply -> verify -> archive
```

它治理一个软件 change；除非有显式的版本化迁移，否则仍是 0.x 系列交付的
公共架构。它是实现产品目的的手段，而不是产品目的本身。

宿主 goal、task 或 plan 可以串联多个 change。SteadySpec 保存每个 change
自身的记录并汇总策略信号，但不定义 goal 到 change 的血缘或完成语义，也
不拥有、认证或保证宿主 goal 状态。

### 实现与 assurance 机制

skills、上下文考古、grill、debate、方向图、证据合同、proof、cross-review、
closure、protocol、checker 和 runtime adapter 支持参考方法的不同部分。
移除或替换其中一个不一定改变稳定核心，但它所覆盖的失败模式和迁移不能
静默消失。

v0.6 closure engine 和实验性 v0.7 assurance protocol，是 `verify`、交接、
真实收口与归档就绪下面按风险启用的可选声明完整性支持。Protocol
conformance 比 SteadySpec 产品或方法 conformance 更窄，也不是方法或五动词
的继任者。

## 有效性与证据边界

SteadySpec 尚未证明相对强 Agent 基线存在因果改善。当前实现、conformance
和 dogfood 证据主要局限在已经声明的单操作者软件环境。迁移到团队、非软件
领域或显著不同的权责模型，仍是未经验证的假设。

全产品对照实验必须尽量固定 Agent、模型版本、工具、权限、仓库快照和任务，
并测量静默目的丢失、无支持的完成声明、盲评产出质量、人的主动介入时间、
恢复成功率/成本及总任务成本。仓库 validator 通过不能回答这个问题。

## 演进与授权边界

修改稳定核心或当前参考架构，都是高风险、由人负责的产品决策。在被表述为
产品方向前，必须具备：

1. 产品合同版本升级；
2. 显式的人类决策记录；
3. 新旧失败模式与覆盖关系图；
4. 兼容、弃用或迁移方案；
5. 证据边界和未解决风险记录；
6. 保留旧合同的 changelog 与发布证据。

Agent、reviewer、benchmark、debate 和多 Agent 共识可以提出这种修改，但
不能批准。确定性验证只让声明出来的合同和架构变化可见；它不能认证人、证明
批准充分知情、判断新设计更好或转移责任。

## 明确不作出的声明

SteadySpec 不承诺无偏输出、全局最优、语义正确、安全、独立审查、充分知情、
法律充分、外部采用，或没有人/Agent 错误；也不证明当前机制是最佳架构。
它的声明更窄：这些是治理委托的显式原则和当前手段，证据有边界，责任决策
保持可见。
