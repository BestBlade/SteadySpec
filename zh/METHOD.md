# SteadySpec 方法

SteadySpec 是一个反漂移方法。v1 软件包把它应用在了软件 SDD 上，但方法本身关乎所有长时间的工作——那些意图、决策、产出、验证和最终记录会相互偏离的工作。

核心主张很简单：工作失败通常不是因为没人写过意图。工作失败是因为后来的执行悄悄地编辑了意图，决策失去了归属者，验证被误当成真相，最终记录被清理干净了，以至于它不再描述实际发生过什么。

本文档是这套方法的可移植版本。`recipes/software-sdd.md` 是软件参考实现。

## 工作是怎么漂移的

不管自费还是公司发的额度，大家都用上 Agent 写代码了。

很快规律就出来了。Agent 写单个小任务很强。但只要你不加限制，vibe coding 式地一直往下聊，对话一长，产出就开始不稳定。不知道哪一轮，Agent 悄悄改了你根本没让它碰的东西。它替你做了一个判断，而做判断的时候你不在场。

于是文档驱动开发被从架子上拿了出来。思路很简单：Agent 写一行代码之前，先聊清楚需求。锁定方向。把规格书写下来。交给 Agent，说：照着来。这套做法，AI 编程圈里大概叫 L3，是第一次有人认真尝试给这件事套上围栏。

**→ 意图先于生产（1）。** 那份规格书就是意图记录——开工前写好的一张便条，说清你要达成什么、边界在哪、完成需要什么证据才算可信。当 Agent 后来碰了边界外的东西，你手里有个东西可以指。没有它，边界只在你脑子里。

L3 管用了一阵。然后就不管用了。

代码从 Agent 手里涌出来，快过任何人能读的速度。瓶颈换到了你身上：你既要审文档，又要看代码，"做完了"到"下一个"之间几乎不留空隙。你的大脑编译不完每一个被改动的文件。

**→ 生产伴随就近检查（4）。** 审查一堆做完了的改动，太晚了。第五个改动可能已经悄悄推翻了第一个改动依赖的前提。没人注意到——因为没人回去重查。解法不是"审查得更仔细"，是在每一个最小单元被产出的那一刻就做检查。一个单元，一次检查，记录结果，然后再往下走。

然后还有一件事。Agent 几乎从不会主动考虑这个新需求跟你的系统有什么联动。它只找最快的路径，做完就行。久而久之，系统里堆满了散落在随机文件中的一次性函数、用三种方式重复实现的逻辑、只因为"Agent 放那儿了"才在那里的东西。

**→ 持久记录同步（6）。** 代码变了。文档没变。每一次漏掉的更新都在悄悄侵蚀信任，直到某天再也没人去读文档——他们知道文档在说谎。解法不是加一句"别忘了更新文档"。解法是设一道门：关掉一个变更之前，扫描哪些文档引用了被改动的代码，必须更新的那些如果没更新，就不让过。

到了这里，循环收紧了。AI 写，AI 做，AI 审。人变成了无情的 Yes 机器——批得太快、太多、太浅。一天交付数个需求之后，你连今天早上的代码长什么样都想不起来了。

**→ 产出对照意图审查（5）。** 审查就退化成了"代码看起来好不好？"，而不是"我们到底有没有解决当初打开这个变更想解决的问题？"你得回到*最初那条意图*——一行代码都没写之前的那条——去对照。不是对照测试。不是对照演进后的规格书。是那条最原始的意图。

到了这个阶段，Agent 就像一个每天早上重新入职的新人。能力强，但项目记忆为零。不知道以前做过什么，也不知道怎么做对这个具体的系统来说算正确。每次新开一个对话，你都得从头教。

**→ 上下文先于自信（2）。** Agent 看到一段看不明白的代码，说"这像死代码，删了吧"——在它动手之前，你需要把已确认的、推断的、未知的分开。别让"大概是"在记录里慢慢变成"肯定是"。没人写下来那个 workaround 为什么存在的时候，正确的答案是"不知道，先标记"——不是"看着多余，删了"。

**→ 决策压力先于共识（3）。** 两句话就达成一致，感觉很高效。经常只是人和 Agent 共享了同一个盲区。锁定方向之前，得有人问一句："什么会搞砸这个方向？如果真搞砸了，有什么替代方案更扛得住？"没被压测过的共识，不过是彼此客气。

到这里，一个新思路出现了：给工作流本身套上显式边界。SDD 告诉 Agent 要建什么。这一层治理告诉 Agent 工作必须怎么被治理——每一步在宣布完成之前需要什么证据、哪些记录必须更新、哪些门必须过。

**→ 不漂白的归档（7）。** 关闭工作是整个过程中最危险的时刻。谁都想讲一个干净的故事。归档想说"做完了"——它不想提那条回退的移动端路径、那个没测的边界场景、那个被推迟的决策。但干净的归档是复利的谎言：下一个人信任这份归档，然后在一个不完整的基础上继续往下盖。归档必须保留实际发生的事。

**→ 局部信号到战略（8）。** 十个变更交付了，其中三个带着一模一样的债务："手工验证待补"。每一个单独拎出来看，当时都是一次合理的局部取舍——"回头再补，先发版"。但三次相同的推迟不是三次取舍，是一种模式。只看单个变更，看不出模式。系统需要帮你跨变更去看。

---

这八个节点不是一张随便排的列表。它们是一条时间线——一份工作从开始到结束的完整生命周期：

```
意图(1) → 上下文(2) → 决策(3) → 生产(4)
    → 对照意图审查(5) → 文档同步(6) → 归档(7)
                                        ↓
                            模式汇总 → 战略(8)
```

不写意图(1)，后来就没法对着它审查(5)。不做审查(5)，就不知道文档该更新什么(6)。不做文档同步(6)，归档写的就是一个已经不存在的世界(7)。归档撒谎(7)，你就永远看不出那些需要战略关注(8)的模式。

每一条抓的是前一条抓不住的漂移。

上面这个故事恰好发生在写代码的场景里。但底层的模式是一样的——意图会漂、决策会没人认领、记录会慢慢跟现实对不上——不管你在做的事情是起草一份合同、写一篇论文、还是列一份小说大纲。名词换掉，漂移不变。

## 关键术语

这几个词会在后面的机制里反复出现。不是学术黑话。每个词命名的，正好是"它不在的时候漂移就从这里进来"的那个东西。

| 术语 | 直白的意思 |
|---|---|
| **意图记录** | 一份简短笔记，工作开始前写好。说清要干什么、边界在哪。就是后来你指着说"这不在计划内"的那张纸。 |
| **可观测检查** | 一个看得见摸得着的信号，证明某件事确实发生了。比如通过的测试、命令行跑出来的结果、一个人的审查确认。不是"我觉得没问题"。 |
| **产出-意图对照** | 做完了之后，拿着最初那条意图去对比做出来的东西。不是比代码好不好看。是问一句"我们到底有没有解决当初想解决的那个问题？" |
| **人属决策** | 关于价值、风险、优先级、方向的决定。流程可以记下来，但不能替你做。只有你能说"这笔债可以接受"或者"这个风险我来扛"。 |
| **漂移** | 意图和实际之间那个静悄悄的偏差。不是 bug，不是谁犯了错。就是工作最自然的状态——没人盯着边界的时候——一定会发生的缓慢的偏离。 |

其余术语（工作介质、工作单元、最终记录）放在各条机制的正文中，用到的地方自然就懂了。

## 1. 意图先于生产

### 机制

在产出之前，先把意图写成之后能被检查的形式。意图记录不需要很长。它需要说清楚：期望的变更、工作的边界、完成前应有的证据、以及停止条件。

这防止最普遍的漂移：产出堆积，然后团队逆向编造一个故事，让产出看起来像是原本就打算做的。

### 软件 SDD 示例

`steadyspec-propose` 在 `steadyspec-apply` 改代码或文档之前，先记录变更意图、证据计划、停止条件和实现边界。

### 可迁移形态

开始之前：
1. 说明意图产出。
2. 说明什么不在工作范围内。
3. 说明哪些可观测检查能证明完成是可信的。
4. 说明何时停止或重新打开意图。

## 2. 上下文先于自信

### 机制

当历史不清楚时，在提议或改变方向之前先恢复已确认的上下文。未知的历史必须标为未知。不要将似是而非的记忆变成永久的"事实"。

这防止上下文漂移：团队以为大家都记得当初为什么做那些选择，但记录已经不再支持这种自信。

### 软件 SDD 示例

`steadyspec-context-archaeology` 在提案、辩论或实现之前，分离出：已确认历史、缺失上下文、过时假设、以及约束。

### 可迁移形态

依赖历史之前：
1. 列出已确认事实，附带来源链接。
2. 单独列出未知项。
3. 将已确认历史转化为约束。
4. 别让猜测进入持久记录。

## 3. 决策压力先于共识

### 机制

锁定方向之前，压测决策树。一次只问一个难题。当存在竞争方向时，运行结构化辩论。要求构造破坏场景、替代方案和盲区检查。

这防止共识漂移：团队把快速达成一致和经过检验的一致搞混了。

### 软件 SDD 示例

`steadyspec-grill` 硬化尚未解决的设计分支。`steadyspec-debate` 分离提议者、挑战者、仲裁者和外部盲区检查，同时声明运时的隔离限制。

### 可迁移形态

达成共识之前：
1. 问：什么会破坏当前方向？
2. 问：在那种失败下，什么替代方案更好？
3. 区分局部反对和战略性反对。
4. 把未解决的人属决策明确记下来，不要藏在流程话语里。

## 4. 生产伴随就近检查

### 机制

产出最小工作单元，运行距离最近的、可信的可观测检查，在继续下一步之前先记录结果。如果产出暴露了意图本身是错的或不完整的，停下来更新意图，而不是悄无声息地扩大工作范围。

这防止执行漂移：生产过程改了目标，但记录还在声称原来的目标被遵循了。

### 软件 SDD 示例

`steadyspec-apply` 定义证明信号、做出最小实现切片、运行检查、记录证据、在检测到漂移时停下来。TDD 是可观测检查是自动化测试时的特例。

### 可迁移形态

生产过程中：
1. 以小单元工作。
2. 每个单元绑定一个可观测检查。
3. 明确记录通过、失败、回退或阻塞。
4. 当检查显示意图需要修正时停下来。

## 5. 产出对照意图审查

### 机制

产出之后，对照原始意图审查产出。不要用好看的产出、通过的检查或审查人的偏好来替换这个问题：产出是否满足了意图和边界？

这防止验证漂移：证据证明了 A，但团队用它来宣称证明了更宽泛的 B。

### 软件 SDD 示例

`steadyspec-review-against-intent` 检查实现是否与意图一致、证据缺口、残余债务，以及是否可以开始文档同步或归档。

### 可迁移形态

接受产出之前：
1. 将产出与每个意图点对齐。
2. 将产出与每个声明的边界对齐。
3. 将证据缺口与缺陷分开标记。
4. 不要把回退当证据。

## 6. 持久记录同步

### 机制

当产出改变了"什么是真的"时，在下一个阶段继续之前先更新持久记录。记录应该说清楚：实现了什么、没实现什么、什么变了、什么仍不确定。

这防止记录漂移：工作是真实的，但持久记录一直在描述之前的世界。

### 软件 SDD 示例

`steadyspec-doc-sync` 在审查之后、归档之前，把实现了的事实写回规格书、变更记录、任务或项目本地文档。

### 可迁移形态

继续之前：
1. 找出未来工作者会信任的持久记录。
2. 用"实现了的事实"更新它们。
3. 区分：已完成工作、已接受债务、回退、未解决风险。
4. 为证据留下源链接。

## 7. 不漂白的归档

### 机制

关闭工作是高风险时刻。人们想要干净的故事。最终记录必须抵抗这种压力，保留实际发生的事——包括未完成项、回退决策和人属风险接受。

这防止归档真相漂移：归档变成了成功叙事，而非可复用的记录。

### 软件 SDD 示例

`steadyspec-archive` 在关闭变更之前，验证：人属决策记录、审查状态、文档同步、已接受债务、以及后续触发项。

### 可迁移形态

声明工作结束之前：
1. 列出所有声称已完成的项目，验证证据。
2. 区分流程产出和人属决策。
3. 标记回退为残余风险，不是验证通过。
4. 为未解决的债务保留归属人或后续触发。

## 8. 局部信号到战略

### 机制

重复发生的局部摩擦不总是局部问题。当同一种漂移跨工作项反复出现时，把它汇总成一个供人审查的战略信号，而不是反复修复症状。

这防止策略漂移：局部补丁堆成山，而真正的运行模型一直是错的。

### 软件 SDD 示例

`steadyspec-strategy-rollup` 收集重复的局部信号、提议文档更新、把价值判断移交 `steadyspec-human-decision-record`。

### 可迁移形态

当模式重复出现时：
1. 归类重复发生的失败或例外。
2. 说出它们所挑战的那个更大的假设。
3. 建议一个人属决策或一个流程更新。
4. 不要把一次局部事件直接升格为战略——除非有多次证据。

## 在软件之外使用这套方法

把 SteadySpec 适配到一个新领域，先映射术语：

1. 在这个领域，什么是意图记录？
2. 什么是工作介质？
3. 什么是工作单元？
4. 什么算作可观测检查？
5. 谁来拥有价值或风险决策？
6. 工作结束后，未来工作者会信任什么记录？

然后"抄袭"八条机制，不要抄软件名词。一篇研究论文、一份合同、一个产品概要、一份小说大纲——都可以用这套反漂移形状，只是用不同的记录和检查。
