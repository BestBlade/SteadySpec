# SteadySpec 范围和边界

SteadySpec 包含 [METHOD.md](METHOD.md) 中的反漂移方法、实验性的 v0.7
Assurance Protocol Candidate，以及 legacy 五动词软件技能包。这些表面的
证据与采用边界不同。

## v0.7 assurance protocol 边界

协议提供领域中性的词汇和确定性 trace 行为，但没有证明它对所有领域都
有效。它绑定精确 target、candidate、evidence、assessment、invalidation、
invocation 与声明出来的 decision。最强状态只是 `ready-for-human`，不是
接受或外部行动授权。

v0.7 的证据覆盖参考进程、可替换实现的黑盒 conformance，以及故意写坏的
always-ready 实现。它不覆盖真实漂移/人工负担/恢复成本的因果改善、外部
采用、actor 身份认证、远程证明、旧快照回滚防护、团队协作或语义真理。
现有 Codex/Claude workflow 与 v0.6 closure 也没有被宣称为薄适配器。

## v0.7 源码分发边界

v0.7.0 只通过官方 Git 仓库分发，`package.json` 会阻止误发布到 npm。
用户从可信 tag 或 commit 构建本地 tarball 后安装。本地 validator 和 CI
只是有边界的证据，不代表 registry 发布、第三方采用、语义真理或人的
发布授权。

## v0.6 closure 产品边界

v0.6 closure 是 `verify` 下面的可选产品/运行时扩展：它没有重写 [METHOD.md](METHOD.md) 的可移植方法，也不是第六个治理动词。它在 Windows 单用户边界内，用指纹绑定的新鲜 Critic、受约束 Builder、操作员配置的 proof 和新鲜 Evaluator，自动消化重复的低风险修复与复验回合。

自动路径不得拥有或改变范围、缩减需求、公共表面、安全/权限、数据/迁移、proof 策略、债务处置、接受、合并、归档或发布决定。任何这类变化、环境阻塞、证据不足、无法收敛或残余未知都要原样上浮给人。`candidate-ready` 只是当前候选、证据包、已声明上下文和未知项的有界机器就绪；它不是人类接受，也不是“多个 Agent 达成真理”。

v0.6 不承诺 Builder 操作系统沙箱、通用副作用隔离、proof 进程与外界的完整隔离、未提供现实信息的正确性、POSIX 支持、多人协作或 CI 发布门。全新的 `npm pack` 安装 smoke 是最终发布证据的一部分；在它通过之前，源代码树的 validator 和 dogfood 不能被写成完整发布就绪。

## Agent 能力分级

SteadySpec 依赖 Agent 去读 SKILL 描述、匹配用户意图、加载正确的技能、然后照着里面的约束执行。这不是免费的——它取决于 Agent 本身的能力。

| 级别 | Agent | 状态 |
|------|--------|--------|
| **Tier 1** | Claude Opus 4.x 及以上；GPT-5 及以上 | 完全支持。自然语言即可触发技能；约束不需要显式提醒就能遵循。 |
| **Tier 2** | **DeepSeek-V4-Pro**（定义 Tier 2 锚点的可负担开源基线）；Claude Sonnet 4.5 及以上；GPT-4o 级别；同档模型 | **v0.6 主要优化目标。** slash command 和技能描述即可触发技能。MUST 开头的步骤能被可靠遵循。软指令可能需要用户强化一下。 |
| **Tier 3** | Claude Sonnet < 4.5；Claude Haiku；GPT-3.5 及以下；本地 300 亿参数以下的模型；工具调用可靠性有限的模型 | **不作承诺。** 技能可能被加载，但自动触发、MUST 步骤遵循、失败模式自检都不可靠。每轮需要你显式引导，或考虑换一套方法论。 |

分不清你的 Agent 属于哪一级？假设是 Tier 3。

## 什么时候用哪条路径

SteadySpec 提供的是一个光谱，不是一条固定流程。你大多数时间应该待在轻量端。

| 路径 | 什么时候用 | 你怎么做 |
|---|---|---|
| **Vibe 模式** | 一次性实验、随口一问、用完就扔的脚本 | 不调动词。不留记录。就干活。 |
| **Level 0–1** | 小型维护项目、日常修 bug | 轻量任务清单 + 每个任务一份证据。不用跑完整提案/归档。 |
| **Level 2–3**（完整治理流程） | 长生命周期功能、API 变更、数据模型变更、任何你半年后需要记得的东西 | 完整 propose -> apply -> verify -> archive，带漂移门。重路径。 |
| **Level 4** | 成熟系统上反复出现的 agent 工作模式 | 项目本地协议。由 strategy-rollup 信号触发，不是默认开启的。 |

规则：从比你认为需要的更轻一级开始。只有漂移真的出现了才往上升。改个配置文件不需要五个动词和全部门。在琐碎的事情上跑完整流程，是在重要的事情上最快放弃它的方式。

## v0.3 责任边界

v0.3 允许 agent 处理更多低风险操作细节，但不会把高风险归属交给 agent。公共接口、迁移、存储/数据丢失风险、安全或权限边界、删除/移除/收窄、与已接受意图冲突、跨三层以上的变更、改变范围或证明策略的重新切片、以及把债务写成证明的归档声明，都必须路由给用户。

低风险 agent 自主决策可以在报告里折叠，但必须保留在决策归属账本里，能被审计。

## v0.4 能力通道边界

v0.4 增加了一个可选能力通道，用在过早收敛本身就是风险的工作上。触发条件包括方向分叉、证据风险、主线风险、高影响产品或架构选择，或用户明确要求更强的解法搜索。它不应该出现在 typo 修复、日常清理、一次性实验、纯状态报告、本地草稿或简单元数据更新上。

这个通道可以让 agent 展开方向、草拟 evidence contract、压力测试主线选择，并保留 parked 或 rejected 的替代方向。它不会让 agent 独立拥有高风险方向决定。公共表面、数据、安全、迁移、不可逆删除和长期主线决定仍然属于用户，或至少必须进入 must-read。

## 单开发者假设

Legacy 软件技能包是为**每次变更一名作者**设计的。技能里提到的"人"
指未来的你或接手者——一个需要在没有原始上下文时重新评估决策的人。

实际意味着：

- 记录必须独立成立。别假设读的人当时在场。
- 没有多负责人工作流、审查分派、并行作者的冲突处理。
- 没有团队仪式（站会、冲刺规划、跨职能签字）。
- SteadySpec 防的漂移是**时间漂移**（你忘了当初为什么决定 X）和**继承漂移**（接手者面对不清晰的状态），不是**协调漂移**（多个作者同时往不同方向走）。

团队使用不禁止，但没验证过，也不是设计目标。在团队里用，把"未来自己"的框架适配到多人场景的成本你自负。

## SteadySpec 不承诺的事

为了让用户别抱错误的期望：

- **它不会自动检测漂移。** 它给你动词，你怀疑有漂移的时候调。它不在后台蹲着。
- **它不会阻止 Agent 犯错。** 它增加的是——在你调了正确动词的前提下——Agent 在正确时刻停下来的概率。
- **它依赖 Agent 去检测 Agent 自己造成的漂移。** v0.3 增加了信任检查点和风险路由审查，但同 Agent 限制仍然存在。高风险工作建议使用外部审查者。honest-tuning 那次失败——同一个 Agent 既写了违规代码又写了校验器，校验器还过了——就是这个结构性限制的典型案例。
- **它不会给规格书做版本管理。** 那是介质（OpenSpec、纯文档、issue tracker）的事。SteadySpec 在介质外面加了一层漂移防御。
- **它不替代测试。** 测试是可观测检查的一种。SteadySpec 问的是"东西做对了没有"，测试验的是"做出来的东西管不管用"。
- **它不强制被使用。** 你可以装了 SteadySpec 但一次动词都不调。包不会吭声。
- **辅助命令有界，不是新的方法论动词。** 除了 `init` 和 docs 模式
  `check`，CLI 还有 `assurance`、`cross-review`、`closure` 和 `hooks`。没有顶层
  `update`、项目级 `uninstall` 或通用 `status`。升级时固定并验证新的
  源码版本，再预演 `init --force`；卸载按 [QUICKSTART.md](QUICKSTART.md)
  的精确路径清理清单执行。
- **issue-tracker 介质仍是实验性。** v0.4 增加了原生 docs 模式结构；GitHub issues / Jira / Linear 可以作为外部记录使用，但 SteadySpec 还不拥有它们的结构和生命周期。
- **Tier 2 优化目标是移动靶。** 方法依赖 Agent 遵循 MUST 前缀指令和基于技能描述的选择。模型行为随版本变化；没有任何 API 合约能保证今天遵循 MUST 前缀的模型在下次更新后仍然遵守。Tier 2 标注描述的是 SteadySpec 发布时测试过的模型，不是永久的保证。
- **不保证多步骤动词流中原语技能选择的正确性。** v0.3 增加了更明确的 flow contract，但原语选择仍依赖 Agent 可靠性。如果你在 `propose-flow` / `apply-flow` / `verify-flow` / `archive-flow` 中观察到加载了错误的原语，请视为路由失败并报告。
- **`archive-flow` 完整性仍有一部分依赖判断。** v0.3 增加了持久真相门和合同锚点，但还没有完整的机械 archive-schema matcher。
- **能力通道是可选的，不是默认仪式。** 如果它出现在日常小改动上，应该把这视为流程变重的味道。
- **同模型 debate 不是独立验证。** 它能结构化审查并记录限制；高风险 claim 需要独立性时应使用外部审查者。
- **高风险主线决定仍归用户。** Agent 可以草拟 direction map 或建议，但不能私下把高影响方向写成持久真相。
- **定性证据需要来源和覆盖范围标签。** 用户报告、同 agent 审查、人工观察和外部审查都是有用信号，但强度不同于确定性检查。
- **`steadyspec-explore-flow` 的 adopt 信号是注意力输入，不是自动升级治理。** 代码行数增长、变更数增长、模块重复漂移或新结构类别可以提高注意力等级，但不会自动改变项目治理级别。

## SteadySpec 的自治理

SteadySpec 把它的方法用在了自己身上。自我应用的记录存在源仓库的 `.meta/changes/`。**`.meta/` 被 git 忽略，不会发到 npm。** 这意味着：

- 发布的包只有*结果*（动词、原语、方法论、配方），没有构建它的*过程记录*。
- 继承这个仓库的接手者可能看不到 `.meta/`。这是故意的。公开的 CHANGELOG.md 和 `git log` 是持久历史。
- 把 `.meta/changes/` 当成 SteadySpec 的私有介质。**别把它当你的项目介质。** 用文档里推荐的介质。

## Assurance protocol 是否适合

当精确快照要跨 Agent、上下文压缩、中断或数天流转，而过期/伪造 ready
声明的代价较高时，可以 dogfood v0.7 protocol。短任务不要只为仪式感使用。
在对照实验完成前，应同时测量收益与维护协议本身的负担。

## Legacy 软件 recipe 是否适合

SteadySpec 适合你，当**以下全部**成立：

0. **你经历过漂移——不是假设，是真的经历过。** 你交付过什么东西，后来发现 Agent 改了你没让它碰的。或者文档已经跟代码对不上了。或者你批得太快，后悔了。SteadySpec 是给已经知道自己为什么需要它的人准备的。如果你还没疼过，这些东西看起来会像是没有理由的多余工作。收藏一下，疼了再回来看。
1. 你的 Agent 是 Tier 1 或 Tier 2。
2. 你单独工作，或异步协作、没有并行作者。
3. 你预计这份工作活得够久——久到未来的你（或接手者）会需要这些记录。
4. 你愿意在合适的时候显式调用这五个动词。

以上任何一条不成立，legacy 软件 recipe 就不是正确选择。用更轻或不同的
工具；这不自动决定更小的 protocol 表面是否适用。
