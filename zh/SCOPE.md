# SteadySpec 范围和边界

SteadySpec 是 [METHOD.md](METHOD.md) 中描述的反漂移方法的一个参考技能包。它提供四个对外动词（`explore` / `propose` / `apply` / `archive`），每个都是一个小闭环，在内部编排了反漂移机制。为了可靠地兑现这件事，v0.2-alpha 明确了具体的边界。安装前请读。

## Agent 能力分级

SteadySpec 依赖 Agent 去读 SKILL 描述、匹配用户意图、加载正确的技能、然后照着里面的约束执行。这不是免费的——它取决于 Agent 本身的能力。

| 级别 | Agent | 状态 |
|------|--------|--------|
| **Tier 1** | Claude Opus 4.x 及以上；GPT-5 及以上 | 完全支持。自然语言即可触发技能；约束不需要显式提醒就能遵循。 |
| **Tier 2** | **DeepSeek-V4-Pro**（定义 Tier 2 锚点的可负担开源基线）；Claude Sonnet 4.5 及以上；GPT-4o 级别；同档模型 | **v0.2-alpha 主要优化目标。** slash command 和技能描述即可触发技能。MUST 开头的步骤能被可靠遵循。软指令可能需要用户强化一下。 |
| **Tier 3** | Claude Sonnet < 4.5；Claude Haiku；GPT-3.5 及以下；本地 300 亿参数以下的模型；工具调用可靠性有限的模型 | **不作承诺。** 技能可能被加载，但自动触发、MUST 步骤遵循、失败模式自检都不可靠。每轮需要你显式引导，或考虑换一套方法论。 |

分不清你的 Agent 属于哪一级？假设是 Tier 3。

## 什么时候用哪条路径

SteadySpec 提供的是一个光谱，不是一条固定流程。你大多数时间应该待在轻量端。

| 路径 | 什么时候用 | 你怎么做 |
|---|---|---|
| **Vibe 模式** | 一次性实验、随口一问、用完就扔的脚本 | 不调动词。不留记录。就干活。 |
| **Level 0–1** | 小型维护项目、日常修 bug | 轻量任务清单 + 每个任务一份证据。不用跑完整提案/归档。 |
| **Level 2–3**（完整四动词） | 长生命周期功能、API 变更、数据模型变更、任何你半年后需要记得的东西 | 完整 propose → apply → archive，带漂移门。重路径。 |
| **Level 4** | 成熟系统上反复出现的 agent 工作模式 | 项目本地协议。由 strategy-rollup 信号触发，不是默认开启的。 |

规则：从比你认为需要的更轻一级开始。只有漂移真的出现了才往上升。改个配置文件不需要四个动词八道门。在琐碎的事情上跑完整流程，是在重要的事情上最快放弃它的方式。

## 单开发者假设

SteadySpec v0.2-alpha 是为**每次变更一名作者**设计的。技能里提到的"人"（`steadyspec-human-decision-record`、`steadyspec-strategy-rollup`、归档门）指的是**未来的你，或接手者**——一个需要在没有原始上下文的情况下重新评估决策的人。

实际意味着：

- 记录必须独立成立。别假设读的人当时在场。
- 没有多负责人工作流、审查分派、并行作者的冲突处理。
- 没有团队仪式（站会、冲刺规划、跨职能签字）。
- SteadySpec 防的漂移是**时间漂移**（你忘了当初为什么决定 X）和**继承漂移**（接手者面对不清晰的状态），不是**协调漂移**（多个作者同时往不同方向走）。

团队使用不禁止，但没验证过，也不是设计目标。在团队里用，把"未来自己"的框架适配到多人场景的成本你自负。

## SteadySpec 不承诺的事

为了让用户别抱错误的期望：

- **它不会自动检测漂移。** 它给你动词，你怀疑有漂移的时候调。它不在后台蹲着。
- **它不会阻止 Agent 犯错。** 它增加的是——在你调了正确动词的前提下——Agent 在正确时刻停下来的概率。
- **它依赖 Agent 去检测 Agent 自己造成的漂移。** 这套方法要求产出代码的同一个 Agent 也去对照意图验证产出。外部审查者被推荐（SteadySpec 自己开发时也用了），但产品本身不提供。honest-tuning 那次失败——同一个 Agent 既写了违规代码又写了校验器，校验器还过了——就是这个结构性限制的典型案例。
- **它不会给规格书做版本管理。** 那是介质（OpenSpec、纯文档、issue tracker）的事。SteadySpec 在介质外面加了一层漂移防御。
- **它不替代测试。** 测试是可观测检查的一种。SteadySpec 问的是"东西做对了没有"，测试验的是"做出来的东西管不管用"。
- **它不强制被使用。** 你可以装了 SteadySpec 但一次动词都不调。包不会吭声。
- **它不提供 `init` 以外的 CLI 命令。** 没有 `update`、`uninstall`、`check`、`status`。升级靠文件操作和重新 init。要从项目里卸掉，看 [QUICKSTART.md](QUICKSTART.md) 里的手动清理清单。没有卸载命令的原因：一个会在用户文件上动刀的按项目卸载命令，v0.2-alpha 阶段不担这个数据丢失的风险。
- **v0.2-alpha 不把 issue-tracker 当介质。** GitHub issues / Jira / Linear 作介质推迟到 v0.3。
- **Tier 2 优化目标是移动靶。** 方法依赖 Agent 遵循 MUST 前缀指令和基于技能描述的选择。模型行为随版本变化；没有任何 API 合约能保证今天遵循 MUST 前缀的模型在下次更新后仍然遵守。Tier 2 标注描述的是 SteadySpec 发布时测试过的模型，不是永久的保证。
- **不保证多步骤动词流中原语技能选择的正确性。** v0.2-alpha 只在 `explore-flow` 中验证了 F7"信任描述"这个架构假设。如果你在 `propose-flow` / `apply-flow` / `archive-flow` 中观察到加载了错误的原语，视为已知的 F7 假设失效，请报告。
- **`archive-flow` Gate 4（完整性）在 v0.2-alpha 是尽力而为的。** 逐字段的结构校验推迟到 v0.3（需要一个 `archive-schema.md` 和 Gate 4 机械匹配器）。目前 Gate 4 靠 Agent 自己判断哪些 archive.md 字段对当前变更是必需的。不同 Agent、不同轮次，结果可能不一样。
- **`steadyspec-explore-flow` 的治理重评估只提供基线指导。** 定量触发（代码行数翻倍、变更数翻倍、模块重复漂移、新结构类别）需要的基线结构 v0.2-alpha 还没有；推迟到 v0.3。

## SteadySpec 的自治理

SteadySpec 把它的方法用在了自己身上。自我应用的记录存在源仓库的 `.meta/changes/`。**`.meta/` 被 git 忽略，不会发到 npm。** 这意味着：

- 发布的包只有*结果*（动词、原语、方法论、配方），没有构建它的*过程记录*。
- 继承这个仓库的接手者可能看不到 `.meta/`。这是故意的。公开的 CHANGELOG.md 和 `git log` 是持久历史。
- 把 `.meta/changes/` 当成 SteadySpec 的私有介质。**别把它当你的项目介质。** 用文档里推荐的介质。

## SteadySpec 适不适合你

SteadySpec 适合你，当**以下全部**成立：

0. **你经历过漂移——不是假设，是真的经历过。** 你交付过什么东西，后来发现 Agent 改了你没让它碰的。或者文档已经跟代码对不上了。或者你批得太快，后悔了。SteadySpec 是给已经知道自己为什么需要它的人准备的。如果你还没疼过，这些东西看起来会像是没有理由的多余工作。收藏一下，疼了再回来看。
1. 你的 Agent 是 Tier 1 或 Tier 2。
2. 你单独工作，或异步协作、没有并行作者。
3. 你预计这份工作活得够久——久到未来的你（或接手者）会需要这些记录。
4. 你愿意在合适的时候显式调用这四个动词。

以上任何一条不成立，SteadySpec v0.2-alpha 就不是正确选择。用更轻或不同的工具。
